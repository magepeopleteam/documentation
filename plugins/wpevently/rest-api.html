REST API Overview

WpEvently exposes REST API endpoints that allow external applications to read event data, retrieve ticket information, and manage attendees programmatically. The API follows WordPress REST API conventions.

Base URL

  • https://yoursite.com/wp-json/mep/v1/ — Base endpoint for all routes
  • All responses are returned in JSON format
  • Standard HTTP status codes indicate success or failure
  • Supports pagination, sorting, and field filtering
  • Rate limiting follows WordPress REST API defaults

Available Endpoints

  • GET /events — Retrieve a list of all events
  • GET /events/{id} — Retrieve a single event by ID
  • GET /tickets — Retrieve all ticket types
  • GET /tickets/{id} — Retrieve a specific ticket type
  • GET /attendees — Retrieve attendee records

Authentication

API requests must be authenticated using one of the supported methods. Choose the approach that best fits your integration scenario.

Authentication Methods

  • Cookie Authentication — For logged-in admin requests from the same domain
  • Application Passwords — Generate app-specific passwords for external services
  • OAuth 1.0a — For third-party applications requiring delegated access
  • Basic Auth — Simple username/password (use over HTTPS only)
  • Application Passwords are the recommended method for most integrations

Setting Up Application Passwords

  • Go to Users → Profile in the WordPress admin
  • Scroll to the Application Passwords section
  • Enter a name for your application and click Add New
  • Copy the generated password — it is shown only once
  • Use it in the Authorization header: Basic base64(username:password)

Event Endpoints

Detailed reference for retrieving event data through the REST API. Filter events by date, category, status, and more using query parameters.

List Events

  • GET /wp-json/mep/v1/events — All published events
  • ?category=workshop — Filter by category slug
  • ?start_date=2025-01-01 — Events starting after this date
  • ?end_date=2025-12-31 — Events starting before this date
  • ?per_page=20 — Results per page (default: 10, max: 100)
  • ?page=2 — Paginated results page number

Single Event Response

  • id — Unique event ID
  • title — Event title (rendered and raw)
  • date — Event start date and time
  • end_date — Event end date and time
  • location — Venue address and coordinates
  • ticket_types — Array of available ticket types

Ticket & Attendee Endpoints

Access ticket type data and attendee records. These endpoints require authenticated requests with appropriate user permissions.

Ticket Endpoints

  • GET /wp-json/mep/v1/tickets — All ticket types across events
  • GET /wp-json/mep/v1/tickets/{id} — Single ticket type details
  • GET /wp-json/mep/v1/events/{id}/tickets — Tickets for a specific event
  • Response includes: name, price, capacity, availability, description
  • Price returned as both formatted string and raw numeric value

Attendee Endpoints

  • GET /wp-json/mep/v1/attendees — All attendee records
  • GET /wp-json/mep/v1/attendees/{id} — Single attendee details
  • GET /wp-json/mep/v1/events/{id}/attendees — Attendees for an event
  • Response includes: name, email, ticket type, purchase date, status
  • Requires at least edit_posts capability

Integration Use Cases

Practical examples of how developers use the WpEvently REST API to build external applications and automate workflows.

External Calendars

  • Sync events to Google Calendar, Outlook, or iCal
  • Poll the events endpoint periodically for new and updated events
  • Map event date, location, and description to calendar event fields
  • Handle recurring event expansion for multi-date sync
  • Implement webhook-style polling for near-real-time updates

CRM & Marketing Automation

  • Export attendee data to HubSpot, Salesforce, or Mailchimp
  • Trigger email campaigns based on event registration data
  • Segment attendees by ticket type, event category, or purchase date
  • Automate post-event follow-up and feedback collection
  • Build custom dashboards with real-time event and sales data

Error Handling & Rate Limiting

Understand how the API communicates errors and how to handle them gracefully in your integration code.

Error Responses

  • 400 Bad Request — Invalid parameters or malformed request
  • 401 Unauthorized — Missing or invalid authentication
  • 403 Forbidden — Authenticated but insufficient permissions
  • 404 Not Found — Resource does not exist
  • Error responses include a message and code field

Best Practices

  • Always implement exponential backoff for retries
  • Cache API responses to reduce server load
  • Use the X-WP-Total and X-WP-TotalPages headers for pagination
  • Handle network timeouts gracefully with user-facing messages
  • Log API errors for debugging and monitoring
